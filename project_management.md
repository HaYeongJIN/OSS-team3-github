# 0.개요
이 문서에서는 github에서 프로젝트를 관리하는 방법에 대해 알아보겠습니다.  

github은 프로젝트를 효율적으로 관리할 수 있는 도구가 여러가지 존재합니다. 그 중 **Issue**, **Wiki**, **Project** 3가지를 중점적으로 다룰 예정입니다.

---

# 1.Issue
**Issue**는 프로젝트의 모든 작업을 관리하는 허브입니다. 버그 리포트 뿐만 아니라 기능 제안이나 회의 안건, 자료 조사등 모든 업무를 *티켓(Ticket)* 단위로 관리합니다. **Issue**의 가장 큰 특징은 비정형적 논의의 정형화입니다.
 해당 기능을 이용하여 팀원들은 **Issue**내 댓글기능을 마크다운으로 이미지나 코드등을 첨부하여 의견을 나눌 수 있고, 모든 과정을 검색하거나 필터링할 수 있습니다.  

 **Issue**는 담당자가 반드시 배정되어야하며, **Labels**를 통해 업무의 성격(속성, 우선순위, 상태등)에 따라 분류할 수 있습니다. **Milestone**은 **Issue**들을 하나의 그룹으로 묶어 진척도를 보여줍니다.
  전체 **Issue** 및 **PR** 개수 대비 완료된 **Issue** 및 **PR** 개수로 나타냅니다. 또한 **Issue**는 키워드를 사용하여 **Pull Request** 와 연동할 수도 있습니다.
 키워드를 사용한 뒤, Merge Pull Request가 완료되면 자동으로 **Issue**가 닫히게 됩니다. *"키워드 #이슈번호"* 의 형식으로 사용할 수 있습니다.

 | 키워드 | 사용상황 | 예시 |
 |---|---|---|
 | Fixes | 버그를 수정했을 때 | Fixes #10 |
 | Closes | 일반적인 할 일을 완료했을 때 | Closes #7 |
 | Resolves | 복잡한 문제나 기능 요청을 해결했을 때 | Resolves #23 |

 # 2.Wiki
 **Wiki**는 프로젝트 관련 정적 정보와 공식 문서를 작성하고 관리하는 공간으로 팀의 '지식 베이스'라고 할 수 있습니다. README는 전체 내용을 간략화한 요약본이고 Wiki는 상세 내용이 모두 기록되어있는 매뉴얼북이라는 차이점이 있습니다. 또한 **Wiki** 자체도 '.wiki.git'이라는 별도의 Git 저장소로 관리됩니다.  
 **Wiki**에 작성하는 내용으로는 크게 4가지가 있습니다.  
 ## 1.Onboarding(온보딩 가이드)  
 신규팀원을 위해 개발 환경 세팅이나 팀 내부 규칙등을 기술한 문서입니다.
 ## 2.Decision Making(의사결정)  
 아키텍처 결정기록(ADR), 회의록등을 작성하는 문서입니다.
 ## 3.Standardization(표준화)  
 코딩 컨벤션, API 디자인 가이드등 협업을 위한 표준화 문서입니다.
 ## 4.Manual(매뉴얼)  
 API명세서, 배포 프로세스, 장애 대응 절차등을 기록한 문서입니다.

 # 3.Project
 **Project**는 **Issue**와 **PR**을 *카드*형태로 시각화하여 작업 흐름과 상태를 관리하는 기능입니다. **Project**는 **Board**, **Table**, **Roadmap** 세 가지 View로 볼 수 있습니다.  
 **Project**에서는 **Issue**와 달리 원하는 데이터 필드를 직접 만들 수 있습니다. P1, P2, P3와 같이 우선순위를 나타낼 수 있고, 해당 작업을 끝내는 데 까지 걸리는 시간을 입력할 수도 있습니다. 또한, 단순히 Open/Closed외에도  Testing, Code Review등 세부 상태를 정의 할 수도 있습니다.

 ## 1.Board View
 **Board View**는 **To Do**, **In Progress**, **Done**등의 칸반보드 형태로 나누어져 있습니다. 현재 해당 안건이 어느 상태에 있는지 카드형태로 나타내어 보여지므로 현재 어떤 상태에 있는지 알기 쉽습니다.

 ## 2.Table View
 엑셀이나 스프레드시트와 같은 형태로 수많은 **Issue**를 한꺼번에 보거나 필터링하는데 유용한 형태입니다. 이러한 특성으로인해 대량의 데이터 분석 및 이슈 관리에 유용하다는 장점이 있습니다.

## 3.Roadmap View
시간순서대로 작업 기간을 막대 그래프로 나타내는 간트 차트(Gantt chart)형태입니다. 프로젝트의 전체 일정관리나 현재 진행상황을 파악하는 데 유용합니다.
